## Context

当前应用使用固定的 CSS 变量和单一的 768px 媒体查询断点来处理响应式布局。侧边栏在移动端通过隐藏文字、缩小宽度的方式适配，但这导致导航体验不佳。各模块的卡片、表单控件在小屏幕上虽然有基础调整，但仍存在内容溢出、间距不合理等问题。

技术栈：原生 JavaScript + CSS3，无框架依赖，使用 Vite 构建。

## Goals / Non-Goals

**Goals:**
- 在 375px - 768px 屏幕范围内提供良好的移动端体验
- 提供 80% - 150% 的全局缩放能力，步进 10%
- 缩放设置在页面刷新后保持
- 不破坏现有桌面端的布局和交互
- 保持代码简洁，避免引入额外依赖

**Non-Goals:**
- 不支持平板横屏的特殊优化（使用桌面端布局即可）
- 不实现响应式字体大小（使用固定 rem 单位）
- 不支持无限缩放（限制在 80% - 150% 范围）

## Decisions

### 1. 移动端导航方案：汉堡菜单 + 抽屉式侧边栏

**决策**：在 ≤768px 时，将侧边栏改为可收起的抽屉式菜单，通过汉堡按钮控制显示/隐藏。

**理由**：
- 当前的图标导航（隐藏文字）在移动端不够直观
- 抽屉式菜单可以完整展示导航文字，提升可用性
- 符合移动端常见的交互模式，用户学习成本低

**替代方案**：
- 底部导航栏：会占用垂直空间，且 4 个导航项在小屏幕上可能拥挤
- 保持当前图标导航：用户体验较差，不推荐

**实现细节**：
- 添加汉堡按钮（固定在左上角）
- 侧边栏默认隐藏，点击按钮后从左侧滑入
- 点击遮罩层或导航项后自动关闭
- 使用 CSS transform 实现动画效果

### 2. 全局缩放实现：CSS transform scale

**决策**：使用 `transform: scale()` 应用在 `#app` 根元素上实现全局缩放。

**功能定位**：
- **主要用途**：放大辅助（100%-150%），方便视力不佳或高分辨率屏幕用户看清内容
- **次要用途**：轻微缩小（80%-100%），在桌面端获得更多可视空间
- **不适用场景**：解决小屏幕适配问题（应通过响应式布局解决）

**理由**：
- 简单直接，一行 CSS 即可实现全局缩放
- 兼容性好，支持所有现代浏览器
- 不触发重排，性能较好

**重要限制**：
- `transform: scale()` 只改变视觉大小，不改变元素占用的布局空间
- 缩小时（<100%）会在页面周围产生空白区域，而不是让内容更紧凑
- 因此缩小功能的实用性有限，主要价值在于放大

**替代方案**：
- 调整 CSS 变量（如 `font-size`）：需要大量修改现有样式，维护成本高
- 使用浏览器原生缩放（`zoom`）：非标准属性，兼容性差，且会改变布局流

**实现细节**：
```css
#app {
  transform: scale(var(--app-scale, 1));
  transform-origin: top left;
}
```

### 3. 缩放控制 UI：侧边栏底部按钮组 + 精确输入

**决策**：在侧边栏底部（版本号上方）添加缩放控制按钮组（- / 100% / +），支持多种调节方式。

**交互方式**：
1. **按钮点击**：`-` 和 `+` 按钮以 10% 步进调整（80% ↔ 150%）
2. **精确输入**：点击中间的百分比按钮，弹出输入框，可输入 80-150 之间的任意整数
3. **滚轮微调**：鼠标悬停在百分比按钮上时，滚轮可以 1% 步进微调

**理由**：
- 侧边栏是全局导航区域，放置全局控制符合用户预期
- 底部位置不干扰主要导航
- 三按钮设计保持简洁，同时提供精确控制能力
- 多种交互方式满足不同用户习惯（快速调整 vs 精确设置）

**实现细节**：
- 输入框验证：只接受 80-150 之间的整数，非法输入恢复到当前值
- 滚轮事件：使用 `wheel` 事件监听，`preventDefault()` 防止页面滚动
- 输入框样式：小型浮层，聚焦时自动选中文本，按 Enter 确认，按 Esc 取消

**移动端处理**：
- 在汉堡菜单的抽屉中同样显示缩放控制
- 移动端不支持滚轮微调（触摸设备无滚轮）
- 点击百分比按钮时，使用原生数字键盘输入（`<input type="number">`）

### 4. 响应式断点策略：增加 480px 和 640px 断点

**决策**：在现有 768px 断点基础上，增加 480px（小手机）和 640px（中等手机）两个断点。

**理由**：
- 480px：针对 iPhone SE 等小屏设备优化
- 640px：针对主流手机（375px - 428px）优化
- 768px：平板和大屏手机的分界线

**断点分配**：
- `@media (max-width: 480px)`：极小屏优化（字体、间距进一步缩小）
- `@media (max-width: 640px)`：移动端通用优化（已部分存在）
- `@media (max-width: 768px)`：平板和移动端分界线（已存在）

## Risks / Trade-offs

### 风险 1：transform scale 可能导致模糊

**风险**：使用 `transform: scale()` 放大时，文字和图形可能出现模糊。

**缓解措施**：
- 限制最大缩放为 150%，避免过度放大
- 在缩放后添加 `will-change: transform` 提示浏览器优化渲染
- 测试主流浏览器的表现，必要时调整策略

### 风险 2：移动端抽屉菜单增加交互复杂度

**风险**：新增汉堡菜单和遮罩层，可能引入新的 bug（如滚动穿透、焦点管理）。

**缓解措施**：
- 使用 `position: fixed` + `overflow: hidden` 防止滚动穿透
- 确保键盘导航和屏幕阅读器的可访问性
- 充分测试各种移动设备和浏览器

### 风险 3：localStorage 缩放设置可能丢失

**风险**：用户清除浏览器数据后，缩放设置会丢失。

**缓解措施**：
- 这是可接受的行为，符合用户预期
- 提供明显的重置按钮（100%），方便用户恢复默认

### Trade-off：移动端性能 vs 功能完整性

**权衡**：移动端设备性能较弱，复杂的动画和阴影可能影响流畅度。

**决策**：
- 保留核心动画（如抽屉滑入），但简化次要动画
- 在移动端减少阴影层数和模糊半径
- 使用 `@media (prefers-reduced-motion)` 尊重用户偏好

## Migration Plan

**部署步骤**：
1. 先实现全局缩放功能（独立功能，风险低）
2. 再实现移动端响应式优化（涉及布局调整，需充分测试）
3. 在测试环境验证各模块在不同屏幕尺寸和缩放级别下的表现
4. 部署到生产环境

**回滚策略**：
- 如果发现严重 bug，可以通过 Git 回滚到上一个稳定版本
- 缩放功能可以通过移除相关 CSS 和 JS 代码快速禁用

**兼容性**：
- 不影响现有桌面端用户
- 移动端用户将立即获得改进的体验

## Open Questions

1. **是否需要支持横屏模式的特殊优化？**
   - 当前设计假设移动端主要使用竖屏，横屏使用桌面端布局
   - 如果用户反馈横屏体验不佳，可以后续添加 `orientation: landscape` 媒体查询

2. **缩放控制是否需要快捷键支持？**
   - 可以考虑添加 `Ctrl +` / `Ctrl -` 快捷键
   - 需要注意与浏览器原生缩放的冲突

3. **是否需要在移动端隐藏某些高级功能？**
   - 当前设计保留所有功能，但如果某些模块在移动端体验极差，可以考虑隐藏或简化
