## Context

当前项目已经具备基于 `core/` 与 `modules/` 分层的前端练耳架构：`core/chord-ear-training.js` 负责纯逻辑题库、出题与播放事件构建，`modules/ear-training.js` 负责配置、播放、作答与反馈 UI。音程训练与现有和弦练耳在交互流程上高度相似，但训练维度不同：它不依赖和弦族、张力音与 voicing，而依赖音程定义、方向、题型与干扰项策略。

项目当前已有可复用基础设施：
- `music-theory.js` 已定义基础音程常量与音名转换函数。
- `audio-engine.js` 已支持单音、和弦和事件序列播放。
- 现有练耳 UI 已具备配置、统计、自动播放与答题反馈模式。

本次变更的主要约束是：
- 音程训练必须保持与现有练耳体验风格一致。
- 题库与 UI 需要作为独立模块存在，避免把不同训练模型堆叠到和弦练耳页面中。
- 第一阶段聚焦基础能力，不引入 MIDI、麦克风输入、指板定位判题或复杂难度系统。

## Goals / Non-Goals

**Goals:**
- 新增独立的音程练耳模块入口，并与现有和弦练耳并列。
- 复用现有 `core/modules` 架构，新增音程题库、题目生成与播放逻辑。
- 支持旋律音程与和声音程两种题型。
- 支持固定根音与随机根音、上行/下行/随机方向、预设模板与自定义训练集合。
- 在答题后展示正确反馈，并允许用户试听所有选项对应的音程，形成对比学习闭环。
- 保持实现可测试，核心逻辑集中在 `core/` 并以纯逻辑测试覆盖。

**Non-Goals:**
- 不在本次变更中实现节奏训练、和弦进行听辨或调式听辨。
- 不在本次变更中加入指板联动反馈、MIDI 输入作答或麦克风识别。
- 不实现自适应难度、错题本、长期学习路径管理等更高层训练系统。
- 不改造现有和弦练耳的题库规则，只调整导航与页面组织以容纳新模块。

## Decisions

### 1. 将音程训练设计为独立模块，而不是扩展现有和弦练耳页面
- 选择：新增独立入口与独立 `core/modules` 文件。
- 原因：音程训练的配置维度、题目模型与反馈方式与和弦练耳不同。拆开后更利于后续加入节奏训练、和弦进行听辨等模块，形成统一的练耳体系。
- 备选方案：把音程训练塞入现有和弦练耳页面。
- 未采用原因：页面会混入不同层级的配置项，用户心智负担上升，也会让 core 逻辑难以维护。

### 2. 题库定义采用“音高层 + 语义层”结构
- 选择：新增独立的 `INTERVAL_DEFINITIONS`，每个音程同时记录 `semitones`、基础名称、爵士标签、分类、描述与混淆项。
- 原因：后续既能支持基础音程训练，也能平滑过渡到 `♭9 / 9 / ♯11 / 13` 这类爵士语义，而不需要重写题库模型。
- 备选方案：直接复用 `music-theory.js` 里的 `INTERVALS` 常量作为完整题库。
- 未采用原因：现有常量缺少训练所需的元数据，无法支撑模板、反馈与干扰项策略。

### 3. 题目生成沿用现有练耳的 pool → question 模式
- 选择：新增 `buildIntervalPool`、`createIntervalQuestion`、`buildIntervalPlaybackEvents` 等函数，结构上贴近现有和弦练耳模块。
- 原因：这样可以最大化复用现有状态管理与 UI 渲染模式，也更容易测试。
- 备选方案：把音程训练写成一套完全独立的新状态机。
- 未采用原因：会增加重复代码与维护成本。

### 4. 下行音程按真实向下距离生成，而不是仅反转播放顺序
- 选择：下行题目通过 `targetMidi = rootMidi - semitones` 生成目标音。
- 原因：这样训练的是实际的向下距离感，而不是把上行音程倒放，教学价值更高。
- 备选方案：总是生成上行距离，再根据方向交换播放顺序。
- 未采用原因：无法真实训练“向下听辨”的能力。

### 5. 选项仅从当前题库中取，按顺序排列
- 选择：`buildOptions` 直接返回当前 pool 中的所有音程作为选项，不引入外部干扰项。
- 原因：用户自行配置题库即是在定义训练难度，选项范围与题库完全一致，行为可预期；混淆项策略会在题库之外引入用户未选择的音程，造成困惑。
- 备选方案：为每个音程定义优先混淆集合，生成选项时先取混淆项，再从当前 pool 补足。
- 未采用原因：混淆项会让选项超出用户配置的题库范围，与"题库即训练集合"的心智模型不符，也增加了题目生成逻辑的复杂度。

### 6. 答后试听作为标准反馈能力内建，而不是额外模式
- 选择：在答题后所有选项都可点击试听，使用当前题目的根音与方向生成该选项对应的音程音频。
- 原因：这能把“答错即结束”改成“答后继续比较学习”，显著提升教学价值。
- 备选方案：只允许试听正确答案，或不支持答后试听。
- 未采用原因：无法满足对比学习需求，也削弱音程辨识模块与普通选择题的差异化价值。

### 7. 第一阶段保持钢琴采样音色，并复用现有播放引擎
- 选择：直接复用 `audio-engine.js` 的 Tone.js 播放基础设施，不引入新依赖。
- 原因：项目已有稳定的钢琴采样与回退机制，足以覆盖音程训练。
- 备选方案：为音程训练新增吉他音色或专用 synth。
- 未采用原因：会扩大本次范围，并引入额外调音与资源管理复杂度。

## Risks / Trade-offs

- [音程与爵士标签语义可能混淆] → 先以 `intervalId` 作为判题主键，`jazzLabel` 仅用于展示与模板分类，避免在第一阶段支持多答案语义。
- [答后试听可能让页面交互显得拥挤] → 仅在答题后显示试听按钮，并沿用现有卡片式选项布局，避免题前界面过重。
- [随机根音与随机方向叠加后题目可能过难] → 通过预设模板控制默认配置，并保留固定根音与单方向入门路径。
- [模块增加后导航复杂度上升] → 将音程练耳定位为与和弦练耳并列的独立训练入口，而不是在一个页面中嵌套二级配置。
- [新增 core 逻辑但 UI 手动测试压力增加] → 纯逻辑部分保持在 `core/` 并配套测试，UI 只做最小必要手动验证。
