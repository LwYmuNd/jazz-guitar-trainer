# 和弦图表模块动态化 - 实施指南（更新版）

## 1. 实施策略调整

根据最新讨论，实施策略调整为：

### 1.1 数据来源策略

**采用：引入完整 guitar-chords-db-json 库 + 精选配置**

- 将 `szaza/guitar-chords-db-json` 作为数据源引入项目
- 创建精选配置文件，定义首版展示的和弦子集
- 运行时根据配置动态加载对应和弦数据
- 避免手工提取和维护和弦数据

### 1.2 模块迁移策略

**采用：新旧模块并存，逐步迁移**

- 保留现有 `chord-diagrams.js` 模块代码
- 创建新模块 `chord-library.js`（或 `chord-dictionary-v2.js`）
- 导航栏新增入口，两个模块并存
- 后期通过配置隐藏旧模块

### 1.3 UI 设计原则

- 复用项目现有的 `.card` / `.controls` / `.stats` 等样式类
- 参考 `ear-training.js` 和 `interval-training.js` 的布局模式
- 保持与其他模块一致的视觉语言

## 2. 引入 guitar-chords-db-json 的方案

### 2.1 方案对比

#### 方案 A：作为 npm 依赖安装

```bash
npm install guitar-chords-db-json
```

**优点：**
- 安装简单
- 版本管理清晰
- 可以通过 package.json 锁定版本

**缺点：**
- 该库可能没有发布到 npm（需要确认）
- 整个库会被打包，体积较大

#### 方案 B：作为 Git Submodule

```bash
git submodule add https://github.com/szaza/guitar-chords-db-json.git resources/references/guitar-chords-db-json
```

**优点：**
- 当前项目已经采用这种方式
- 可以选择性加载文件

**缺点：**
- submodule 管理相对复杂
- 需要手动更新

#### 方案 C：直接复制到项目中

将需要的 JSON 文件复制到 `src/data/chords/` 目录。

**优点：**
- 完全控制数据
- 构建简单

**缺点：**
- 失去上游更新能力
- 数据量大时不便维护

### 2.2 推荐方案

**推荐方案：当前已有完整数据源 + 构建时按需加载 + 全局统一 ASCII 命名**

当前项目已经将完整的 `guitar-chords-db-json` 放入 `resources/references/`，包含：

- ✅ 12 个根音目录（C, C#, D, D#, E, F, F#, G, G#, A, A#, B）
- ✅ 每个根音 756 种和弦类型
- ✅ 总计约 9000+ 个 JSON 文件

建议实施方案：

1. 保持当前数据源位置不变
2. **项目全局统一采用 ASCII 升降号命名**（见 2.3 节）
3. 创建构建脚本，根据精选配置提取需要的 JSON
4. 将提取的数据打包到 `src/core/chord-library-data.js`

### 2.3 升降号符号统一策略（重要决策）

**决策：项目全局统一升降号符号为 ASCII 表示**

| 类型 | 旧方案（Unicode） | 新方案（ASCII） | 说明 |
|------|------------------|----------------|------|
| 降号音名 | `D♭`, `E♭` | `Db`, `Eb` | 仍用降号，只改符号 |
| 升号音程 | `♯11`, `♯9` | `#11`, `#9` | 符号改为 ASCII |
| 降号音程 | `♭9`, `♭13` | `b9`, `b13` | 符号改为 ASCII |
| 和弦名 | `7♯9`, `m7♭5` | `7#9`, `m7b5` | 符号改为 ASCII |

**重要说明：**

- **音名选择逻辑不变**：项目内部仍用 `Db`, `Eb`, `Gb`, `Ab`, `Bb`（降号音名）
- **数据源仍用升号**：`C#`, `D#`, `F#`, `G#`, `A#`（升号音名）
- **根音映射仍需要**：`Db ↔ C#`, `Eb ↔ D#` 等
- **只改显示符号**：从 Unicode `♯/♭` 改为 ASCII `#/b`

**为什么这样更合适：**

1. 与数据源、Tone.js、JSON 文件名符号一致
2. 减少显示层的 `♯ ↔ #`、`♭ ↔ b` 转换逻辑
3. 代码更清晰，测试更简单
4. 不影响现有音名选择逻辑（降号调仍用降号）

**修改复杂度评估：**

这个调整**复杂度较低，属于显示层清理**。

需要修改的文件：
- `src/core/chord-ear-training.js` - 移除显示转换逻辑（约 10 处）
- `src/core/interval-ear-training.js` - 移除显示转换逻辑（约 5 处）
- `src/modules/*.js` - UI 文案中的硬编码符号（约 10 处）
- `tests/*.test.js` - 测试断言中的 Unicode 符号（约 20 处）

**不需要修改：**
- `src/core/music-theory.js` - 音名常量定义保持不变
- 音名选择逻辑
- 根音映射逻辑

**实施建议：**

建议把这个调整作为**和弦库新模块开发前的轻量清理**：

1. 先完成符号统一（独立 PR，约 1-2 小时）
2. 再开始和弦库新模块开发
3. 这样新模块从一开始就使用统一的 ASCII 符号

### 3.2 配置说明

- `roots`：支持的 12 个根音
- `chordTypes`：精选的和弦类型列表，每个包含：
  - `suffix`：对应 guitar-chords-db-json 的文件名
  - `category`：所属分类
  - `label`：UI 显示名称
  - `priority`：优先级（1=首版必须，2=首版推荐，3=后续扩展）
- `maxPositionsPerChord`：每个和弦最多展示几个指型
- `positionFilters`：指型筛选规则

## 4. 数据加载与处理流程

### 4.1 构建时处理（推荐）

创建构建脚本 `scripts/build-chord-library.js`：

```js
import fs from 'fs';
import path from 'path';
import { CHORD_LIBRARY_CONFIG } from '../src/config/chord-library-config.js';

const CHORDS_SOURCE_DIR = './resources/references/guitar-chords-db-json';
const OUTPUT_FILE = './src/core/chord-library-data.js';

// 项目内部降号音名 → 数据源升号目录名
function projectRootToDataSourceRoot(root) {
  return CHORD_LIBRARY_CONFIG.rootMapping[root] || root;
}

function loadChordData() {
  const chordData = {};
  
  CHORD_LIBRARY_CONFIG.chordTypes.forEach(({ suffix, category, label, priority }) => {
    // 只加载 priority <= 2 的和弦（首版范围）
    if (priority > 2) return;
    
    CHORD_LIBRARY_CONFIG.roots.forEach(projectRoot => {
      // 映射到数据源目录名
      const dataSourceRoot = projectRootToDataSourceRoot(projectRoot);
      const filePath = path.join(CHORDS_SOURCE_DIR, dataSourceRoot, `${suffix}.json`);
      
      if (fs.existsSync(filePath)) {
        const rawData = JSON.parse(fs.readFileSync(filePath, 'utf-8'));
        const key = `${projectRoot}-${suffix}`;
        
        chordData[key] = {
          root: projectRoot,  // 使用项目内部降号音名
          suffix,
          category,
          label,
          positions: filterPositions(rawData.positions)
        };
      }
    });
  });
  
  return chordData;
}

function filterPositions(positions) {
  const { maxPositionsPerChord, positionFilters } = CHORD_LIBRARY_CONFIG;
  
  // 筛选和排序逻辑
  let filtered = positions.filter(pos => {
    const frets = parseFrets(pos.frets);
    const span = getFretSpan(frets);
    return span <= positionFilters.maxFretSpan;
  });
  
  // 按品位范围排序，优先低把位
  filtered.sort((a, b) => {
    const aMin = getMinFret(parseFrets(a.frets));
    const bMin = getMinFret(parseFrets(b.frets));
    return aMin - bMin;
  });
  
  return filtered.slice(0, maxPositionsPerChord);
}

function parseFrets(fretString) {
  return fretString.split('').map(c => {
    if (c === 'x') return -1;
    if (c >= '0' && c <= '9') return parseInt(c);
    // 处理 10+ 品位（用字母表示）
    return c.charCodeAt(0) - 87; // 'a' = 10
  });
}

function getFretSpan(frets) {
  const playable = frets.filter(f => f > 0);
  if (playable.length === 0) return 0;
  return Math.max(...playable) - Math.min(...playable);
}

function getMinFret(frets) {
  const playable = frets.filter(f => f > 0);
  return playable.length > 0 ? Math.min(...playable) : 0;
}

// 生成输出文件
const chordData = loadChordData();
const output = `// Auto-generated chord library data
// Generated at: ${new Date().toISOString()}
// Source: guitar-chords-db-json

export const CHORD_LIBRARY_DATA = ${JSON.stringify(chordData, null, 2)};
`;

fs.writeFileSync(OUTPUT_FILE, output, 'utf-8');
console.log(`✅ Generated ${OUTPUT_FILE} with ${Object.keys(chordData).length} chord entries`);
```

### 4.2 在 package.json 中添加构建命令

```json
{
  "scripts": {
    "build:chords": "node scripts/build-chord-library.js",
    "prebuild": "npm run build:chords",
    "dev": "npm run build:chords && vite"
  }
}
```

### 4.3 运行时加载（备选方案）

如果不想构建时处理，也可以运行时动态加载：

```js
// src/core/chord-library-loader.js
import { CHORD_LIBRARY_CONFIG } from '../config/chord-library-config.js';

export async function loadChordData(root, suffix) {
  try {
    // Vite 支持动态 import
    const module = await import(
      `../../resources/references/guitar-chords-db-json/${root}/${suffix}.json`
    );
    return module.default;
  } catch (error) {
    console.warn(`Failed to load chord: ${root}${suffix}`, error);
    return null;
  }
}
```

**注意：** 运行时加载需要确保 Vite 能正确处理这些 JSON 文件。

## 9. 关键技术细节

### 9.1 guitar-chords-db-json 数据格式

```json
{
  "key": "C",
  "suffix": "maj7",
  "positions": [
    {
      "frets": "x32000",
      "fingers": "032000"
    },
    {
      "frets": "x35453",
      "fingers": "021341",
      "barres": "3",
      "capo": "true"
    }
  ]
}
```

**字段说明：**
- `frets`：六弦品位，`x` 表示不弹，`0` 表示空弦，数字表示品位
- `fingers`：指法，`0` 表示空弦或不弹
- `barres`：横按品位（可选）
- `capo`：是否需要横按（可选）

### 9.2 数据归一化

需要将 guitar-chords-db-json 格式转换为项目内部格式：

```js
function normalizePosition(rawPosition, root, suffix) {
  const frets = parseFrets(rawPosition.frets);
  const fingers = parseFingers(rawPosition.fingers);
  
  return {
    root,
    suffix,
    frets,
    fingers,
    barres: rawPosition.barres ? parseInt(rawPosition.barres) : null,
    isBarreChord: rawPosition.capo === 'true',
    baseFret: calculateBaseFret(frets),
    mutedStrings: getMutedStrings(frets),
    openStrings: getOpenStrings(frets),
    intervals: calculateIntervals(frets, root, suffix),
  };
}
```

### 9.3 Interval 计算

需要根据和弦类型和品位计算每根弦的音程：

```js
function calculateIntervals(frets, root, suffix) {
  const chordIntervals = getChordIntervals(suffix); // 例如 maj7 = [0, 4, 7, 11]
  const rootSemitone = noteToSemitone(root);
  
  return frets.map((fret, stringIndex) => {
    if (fret < 0) return null; // muted
    
    const stringTuning = GUITAR_TUNING[stringIndex];
    const noteSemitone = (stringTuning + fret) % 12;
    const intervalSemitone = (noteSemitone - rootSemitone + 12) % 12;
    
    return semitoneToIntervalLabel(intervalSemitone);
  });
}
```

## 10. 风险与注意事项

### 10.1 数据体积

- guitar-chords-db-json 完整库包含 9000+ 个 JSON 文件
- 如果全部打包，会显著增加构建体积
- 构建时处理可以有效控制最终打包体积
- 首版建议只加载 priority <= 2 的和弦类型
- 预计首版数据量：约 20-30 种和弦类型 × 12 根音 × 平均 6 个指型 ≈ 1500-2000 个指型数据

### 10.2 数据质量

- 外部数据库的指型质量参差不齐
- 需要通过 `positionFilters` 筛选合理的指型
- 可能需要人工审核部分和弦的指型选择

### 10.3 兼容性

- 确保新旧模块可以并存
- 不要破坏现有模块的功能
- 保持 API 和数据结构的向后兼容

### 10.5 升降号符号一致性

- 项目全局符号已统一为 ASCII（`Db`, `#11`, `b9`）
- 音名仍用降号（`Db`, `Eb` 等），与数据源升号（`C#`, `D#`）需要映射
- 确保新模块从一开始就使用 ASCII 符号
- 如果旧模块尚未统一符号，需要注意新旧模块并存期间的显示差异

### 10.4 性能

- 如果运行时加载大量 JSON，可能影响性能
- 建议采用构建时处理方案
- 考虑懒加载或分批加载策略
