# 和弦图表模块动态化 - 详细设计文档

## 1. 设计目标

将当前基于静态 PNG 图片的和弦图表模块，重构为基于数据驱动的动态 SVG 渲染模块。

### 1.1 核心目标

- 移除对 244 张静态 PNG 图片的依赖
- 实现动态 SVG 和弦图渲染
- 支持根音切换（当前只能看固定根音的图）
- 支持多种按法展示（同一和弦显示多个 voicing）
- 统一项目内的和弦图渲染逻辑
- 为后续模块联动打好基础

### 1.2 非目标

首版不做：
- 完整接入 `guitar-chords-db-json` 的 99230 个和弦（数据量过大）
- 用户自定义和弦编辑器
- 和弦指型自动生成算法（使用预定义数据）
- 复杂的和弦搜索与推荐系统

## 2. 当前实现分析

### 2.1 现有代码结构

**当前和弦图表模块** (`src/modules/chord-diagrams.js`)：
- 硬编码 9 个分类，共 244 张静态图片
- 左侧分类导航 + 右侧图片展示
- 点击分类按钮 → 加载对应 PNG 图片
- 无交互能力，无根音切换

**和弦练耳模块已有能力** (`src/modules/ear-training.js:118-181`)：
- 动态 SVG 和弦图渲染函数 `renderDynamicDiagramSVG()`
- 支持显示弦号、品位、音程标签
- 支持 muted strings (X) 和 open strings (O)
- 支持动态 baseFret 计算

**现有 Voicing 数据** (`src/core/chord-voicings.js`)：
- Drop2 / Drop3 数据库（6 种和弦质量 × 3 组弦集 × 4 个转位）
- 支持根音转调 `transposeVoicing()`（用于练耳模块的教学型 voicing）
- 标准调弦 MIDI 值定义

### 2.2 可复用资源

1. **SVG 渲染逻辑**：`ear-training.js` 中的 `renderDynamicDiagramSVG()` 可直接复用
2. **Voicing 数据**：`chord-voicings.js` 中的 Drop2/Drop3 数据库
3. **完整和弦数据库**：`resources/references/guitar-chords-db-json/` 包含：
   - 12 个根音目录（C, C#, D, D#, E, F, F#, G, G#, A, A#, B）
   - 每个根音 756 种和弦类型
   - 每种和弦类型包含多个指型变化
   - 总计约 9000+ 个 JSON 文件

### 2.3 关键发现

这次设计里最重要的技术判断有四个：

1. **完整的 guitar-chords-db-json 数据库已经放入项目**
   - 位置：`resources/references/guitar-chords-db-json/`
   - 包含 12 个根音目录（C, C#, D, D#, E, F, F#, G, G#, A, A#, B）
   - 每个根音下有 756 种和弦类型
   - 总计约 9000+ 个和弦指型 JSON 文件
   - 这为动态和弦库提供了完整的数据基础

2. **项目应统一升降号符号为 ASCII 表示**
   - 数据源使用 ASCII 符号：`C#`, `7#9`, `m7b5`
   - 项目内部音名保持不变（仍用 `Db`, `Eb`, `Gb`, `Ab`, `Bb`）
   - 但显示符号统一改为 ASCII：`Db` 而非 `D♭`，`7#9` 而非 `7♯9`
   - 移除显示层的 Unicode 转换逻辑（`replace(/#/g, '♯')` 等）
   - 根音映射仍然需要（`Db ↔ C#`），但符号统一后代码更清晰

3. **项目已经有动态指板图渲染能力**
   - `src/modules/ear-training.js:118` 开始的 `renderDynamicDiagramSVG()` 已经证明动态渲染方案可行
   - 这意味着和弦图表模块不需要重新发明一套绘图体系

4. **当前最适合的路线是”完整数据源 + 精选配置 + 动态渲染”**
   - 不需要手工维护和弦数据
   - 通过配置文件控制首版展示范围
   - 构建时生成精选数据子集
   - 保持数据完整性和产品可控性的平衡

## 3. 设计原则

### 3.1 数据优先，不再依赖图片

和弦图表模块的核心数据应从“图片文件名”升级为“结构化和弦指型对象”。

当前是：

```js
{id:'maj7', name:'Major 7', img:'maj7.png'}
```

目标是：

```js
{
  id: 'maj7-shell-e',
  category: 'shell',
  family: 'maj7',
  label: 'Major 7',
  rootString: 6,
  shapeSource: 'caged',
  referenceKey: 'C',
  suffix: 'maj7',
  frets: [-1, 3, 2, 4, 0, -1],
  fingers: [0, 2, 1, 3, 0, 0],
  intervals: [null, '1', '3', '7', '5', null],
  tags: ['shell', 'root-position', 'compact']
}
```

### 3.2 首版先做“精选可练”的和弦字典

这个模块的目标不是做成“和弦百科全书”，而是做成“对爵士吉他学习者真正有训练价值的和弦字典”。

所以首版应该优先收录：

- Basic 7th
- Shell voicings
- 常见 Major / Minor / Dominant extensions
- Drop2 / Drop3
- Quartal
- 常见 altered dominant

不应该首版就把大量重复、音乐价值不高、指型体验差的按法全部展示出来。

### 3.3 与练耳模块共享渲染基础设施

和弦练耳和和弦图表不应维护两套独立的指板图渲染逻辑。

目标是抽出共享能力，例如：

- `renderChordDiagramSVG()`
- `normalizeChordDiagram()`
- `getBaseFret()`
- `getMutedStrings()`
- `formatIntervalLabel()`

### 3.4 优先支持“浏览 + 选择 + 对比”三种动作

用户在和弦字典里最常见的动作不是“搜索全宇宙和弦”，而是：

1. 浏览某一类和弦
2. 在同类按法中切换
3. 对比同一和弦在不同把位 / 不同 family 下的差异

所以 UI 应该围绕这三种动作设计。

## 4. 首版功能范围

## 4.1 首版必须支持

### A. 动态 SVG 渲染

- 每个和弦图由数据实时渲染
- 支持品位、弦号、X/O、按点
- 支持在圆点中显示 interval label

### B. 根音切换

- 同一和弦类型支持根音切换
- 例如 `Cmaj7`、`Dbmaj7`、`Dmaj7` ... `Bmaj7`
- 底层通过“项目内部根音 → 数据源根音目录”映射后直接查询对应 JSON 实现

### C. 多个指型展示

- 同一和弦可展示多个可选按法
- 用户能看到“同类 voicing 的不同落点”

### D. 分类浏览

沿用当前左侧分类浏览方式，但升级数据结构。

建议首版保留这些大类：

- Basic
- Shell
- Extended Major
- Extended Minor
- Dominant / Altered
- Drop 2
- Drop 3
- Quartal

### E. 和弦详情信息

每个和弦图下方显示：

- 和弦名
- 和弦构成音 / interval
- 所属类别
- 简单标签，例如 `rootless` / `shell` / `drop2`

## 4.2 首版暂不支持

- 文本自由搜索
- 自动推荐最佳按法
- 收藏夹
- 用户自建和弦库
- MIDI 播放和弦图中的每个按法
- 自动分析手型难度分数

## 5. 数据方案设计

## 5.1 为什么不直接继续使用 PNG

继续用 PNG 的问题：

- 根音无法切换
- 不能显示组成音和 interval
- 不便于复用到其他模块
- 图片维护成本高
- 不利于未来做模块联动

所以静态图片应从“主实现”降级为“迁移期参考素材”。

## 5.2 为什么不直接把 `resources/references` 当运行时数据源

原因有三个：

1. **数据体积过大**：9000+ 个 JSON 文件全部打包会显著增加构建体积
2. **数据质量参差不齐**：外部数据库包含大量重复、低质量或不适合爵士学习的指型
3. **原始数据格式仍需归一化**：即使项目全局统一为 ASCII 命名，外部 JSON 仍然需要筛选、排序、补充标签与 interval 信息

因此更适合的方案是：

**通过构建脚本从完整数据源中提取精选子集，生成项目运行时使用的和弦字典数据。**

## 5.3 推荐的数据分层

建议拆成三层：

```text
resources/references/
  └─ guitar-chords-db-json/        # 外部参考，不直接运行时使用

src/core/
  ├─ chord-dictionary-data.js      # 项目运行时和弦字典数据
  ├─ chord-diagram-renderer.js     # 渲染工具
  └─ chord-dictionary.js           # 数据查询、归一化、过滤逻辑
```

### 第 1 层：原始数据源层

来源：`resources/references/guitar-chords-db-json/`

特点：
- 完整的 12 根音 × 756 和弦类型
- 每个和弦包含多个指型变化
- 使用 ASCII 命名（`C#`, `7#9` 等）

用途：
- 作为构建脚本的输入源
- 对照和验证指型
- 后续扩展时的数据储备

### 第 2 层：运行时数据层

来源：手工整理后的精选数据

用途：
- 作为前端实际展示的数据源
- 数据体积可控
- 命名和标签可完全贴合本项目产品语义

### 第 3 层：查询与归一化层

职责：
- 按分类取和弦
- 将项目内部根音映射到数据源目录名
- 计算 interval label
- 对原始 positions 做筛选、归一化、补充标签
- 生成 UI 需要的标准结构

## 6. 数据结构设计

## 6.1 核心类型

建议统一成以下数据结构：

```js
{
  id: 'Db-maj7-pos-1',
  categoryId: 'basic',
  familyId: 'maj7',
  familyLabel: 'Maj7',
  displayName: 'Dbmaj7',
  variationLabel: 'Position 1',
  rootString: 5,
  frets: [-1, 4, 6, 5, 6, -1],
  fingers: [0, 1, 3, 2, 4, 0],
  intervals: [null, '1', '5', '7', '3', null],
  tags: ['basic', 'compact', 'barre'],
  source: {
    type: 'database',
    ref: 'guitar-chords-db-json/C#/maj7.json',
    rawKey: 'C#',
    projectKey: 'Db'
  }
}
```

## 6.2 字段说明

- `id`：唯一标识
- `categoryId`：模块里的大类
- `familyId`：和弦家族，如 `maj7` / `m7` / `7`
- `displayName`：用于 UI 展示
- `variationLabel`：用于区分同类按法
- `rootString`：该按法主要根音所在弦
- `rootString`：该按法主要根音所在弦
- `source.rawKey`：数据源根音目录名，例如 `C#`
- `source.projectKey`：项目内部显示根音，例如 `Db`
- `frets`：六弦数组，`-1` 代表 muted，`0` 代表空弦
- `fingers`：指法，可选
- `intervals`：六弦对应的音程标签，统一使用 ASCII，如 `b3`、`#11`
- `tags`：用于筛选和展示标签
- `source`：追溯来源

## 6.3 分类模型

建议把现有 `CATEGORIES` 从“图片目录映射”升级为“产品语义分类”：

```js
[
  { id: 'basic', label: 'Basic 7th' },
  { id: 'shell', label: 'Shell Voicings' },
  { id: 'major', label: 'Extended Major' },
  { id: 'minor', label: 'Extended Minor' },
  { id: 'dominant', label: 'Dominant / Altered' },
  { id: 'drop2', label: 'Drop 2' },
  { id: 'drop3', label: 'Drop 3' },
  { id: 'quartal', label: 'Quartal' }
]
```

这样后续新增数据时，不需要再和图片文件名耦合。

## 7. 指型来源策略

## 7.1 首版采用“完整数据源 + 精选筛选”

最推荐的策略：

- 直接使用完整 `guitar-chords-db-json` 作为上游数据源
- 用户切换根音时，直接查询对应根音目录下的和弦 JSON
- 通过构建脚本和配置文件筛选高质量、训练价值高的形状
- 运行时只消费筛选后的项目数据子集

例如：

- 用户要看 `Dbmaj7`
- 数据层通过根音映射找到数据源目录 `C#`
- 读取 `guitar-chords-db-json/C#/maj7.json`
- 从 `positions` 里筛出适合展示的若干按法

## 7.2 为什么这个方案适合当前项目

因为你现在已经有完整数据库，继续走”C 模板 + fret 平移”的路线反而会增加额外复杂度。

这意味着：

- 不需要维护 C 模板根音概念
- 不需要处理空弦和弦转调失真问题
- 可以直接使用数据库里原生存在的各根音指型
- 更符合“完整数据源 + 精选展示”的产品方向

## 7.3 哪些类别直接使用现有项目数据

### Drop 2 / Drop 3

直接使用 `src/core/chord-voicings.js` 里的数据库。

优点：
- 已经和当前项目逻辑一致
- 转位、弦组定义清晰
- 非常适合与练耳模块共享

### Shell / Basic / Extended / Altered / Quartal

建议从：

- 当前 PNG 所代表的类别语义
- `guitar-chords-db-json/<root>/*.json` 完整数据源
- 你自己的产品判断

中整理出一份 curated dataset。

也就是说，**不是自动全量搬运到前端，而是基于完整数据库做精选。**

## 8. 渲染层设计

## 8.1 抽取共享渲染模块

建议新增：

```text
src/core/chord-diagram-renderer.js
```

职责：
- 把标准化和弦图对象渲染成 SVG
- 提供统一渲染 API
- 被 `ear-training.js` 和 `chord-diagrams.js` 共用

## 8.2 推荐 API

```js
renderChordDiagramSVG(diagram, options)
normalizeChordDiagram(rawShape)
getDiagramBaseFret(frets)
getMutedStrings(frets)
getOpenStrings(frets)
```

### 输入结构

```js
{
  title: 'Dmaj7 Shell',
  frets: [-1, 5, 4, 6, -1, -1],
  intervals: [null, '1', '3', '7', null, null],
  fingers: [0, 2, 1, 3, 0, 0]
}
```

### 输出

- SVG markup string

## 8.3 Base Fret 计算规则

为了显示高把位和弦，需要统一 base fret 逻辑：

- 如果所有按弦 fret 都在 `0~4`，则 `baseFret = 1`
- 如果最高和最低可按 fret 差较小且整体位置高于 4 品，则显示偏移起始品
- 例如 frets `[8, 10, 9, 9, 8, 8]` → `baseFret = 8`

## 8.4 Interval Label 规则

建议默认显示 interval，而不是音名。

例如：
- `1`
- `3`
- `5`
- `7`
- `9`
- `b9`
- `#11`

原因：
- 更符合爵士和声学习语境
- 与练耳模块现有视觉语言一致
- 对根音切换更友好

后续可扩展一个切换：
- 显示音程
- 显示音名

但首版先不做切换。

## 9. 查询与归一化逻辑设计

## 9.1 推荐的查询入口

新增：

```text
src/core/chord-dictionary.js
```

职责：
- 获取分类列表
- 获取某类和弦数据
- 根据项目内部根音映射到数据源目录
- 输出渲染所需标准数据

## 9.2 推荐函数

```js
getChordCategories()
getChordFamiliesByCategory(categoryId)
getChordShapes(categoryId, familyId, root)
mapProjectRootToDataSourceRoot(root)
normalizePosition(rawPosition, meta)
shapeToDiagram(shape, root)
```

## 9.3 根音查询规则

给定项目内部根音和目标和弦类型：

1. 将项目内部根音映射到数据源目录名，例如 `Db -> C#`
2. 读取对应目录下的和弦 JSON，例如 `guitar-chords-db-json/C#/maj7.json`
3. 从 `positions` 中筛选适合展示的若干按法
4. 对筛选结果补充 interval、标签、variationLabel、baseFret 等视图字段

### 关键限制

完整数据库虽然覆盖全面，但存在以下实际问题：

- 某些和弦类型有大量重复或近似指型
- 某些 suffix 对爵士学习价值不高
- 某些 position 可演奏性较差或跨度过大
- 同名和弦下可能混有 slash-like / bass variation 文件

因此首版应把重点放在：

- 直接查询完整数据库
- 通过配置和规则筛选高质量 positions
- 不把数据库原始内容原样暴露给前端

## 12. 实施步骤建议

## Step 1：抽离共享渲染器

从 `src/modules/ear-training.js:118` 附近提取通用 SVG 渲染逻辑。

目标：

## 13. 风险与注意事项

## 13.1 完整数据库中的原始指型质量不稳定

这是这次设计里最大的实现风险之一。

虽然数据库很完整，但原始 positions 并不等于都适合直接展示给用户。

典型问题包括：
- 同一和弦下重复或近似按法很多
- 某些按法跨度过大
- 某些按法虽然成立，但不适合作为现代爵士吉他基础训练素材

### 应对策略

- 首版优先展示筛选后的高质量 positions
- 用配置文件限制 suffix 范围
- 用规则限制 fret span、优先低把位、优先可读性强的形状

## 13.2 不同来源的数据格式不统一

- `chord-voicings.js` 是项目自有结构
- `guitar-chords-db-json` 是外部 JSON 结构
- 当前 PNG 分类又是另一套语义

### 应对策略

必须先做统一 normalize 层，不能把多套结构直接混进 UI。

## 13.3 首版范围容易失控

一旦开始做“动态和弦字典”，很容易想继续加：
- 搜索
- 收藏
- 全量和弦数据库
- 播放
- 难度排名

### 应对策略

首版只解决三件事：
- 动态渲染
- 根音切换
- 多按法浏览

其他全部后置。

## 16. 升降号符号统一策略

项目后续应统一采用 ASCII 升降号符号：

### 统一范围

**只改符号显示，不改音名选择逻辑：**

| 类型 | 旧（Unicode） | 新（ASCII） | 说明 |
|------|--------------|------------|------|
| 降号音名 | `D♭`, `E♭` | `Db`, `Eb` | 仍用降号音名，只改符号 |
| 升号音程 | `♯11`, `♯9` | `#11`, `#9` | 符号改为 ASCII |
| 降号音程 | `♭9`, `♭13` | `b9`, `b13` | 符号改为 ASCII |
| 和弦名 | `7♯9`, `m7♭5` | `7#9`, `m7b5` | 符号改为 ASCII |

**不改变的部分：**

- `NOTE_NAMES_FLAT` 仍然是 `['C','Db','D','Eb',...]`（降号音名）
- 数据源目录仍然是 `C#`, `D#` 等（升号音名）
- **根音映射仍然需要**：`Db ↔ C#`, `Eb ↔ D#` 等

### 为什么这样更合适

1. 与数据源、Tone.js、JSON 文件名保持符号一致
2. 减少显示层的 `♯ ↔ #`、`♭ ↔ b` 转换逻辑
3. 代码更清晰，测试更简单
4. 不影响现有音名选择逻辑（降号调仍用降号）

### 修改复杂度评估

这个调整**复杂度较低，属于显示层清理**。

主要改动：
- 移除 `src/core/chord-ear-training.js` 和 `interval-ear-training.js` 中的 `replace(/#/g, '♯')` 等转换
- 更新测试断言中的 Unicode 符号为 ASCII
- UI 文案中的硬编码 Unicode 符号改为 ASCII

**不需要改动：**
- `music-theory.js` 中的音名常量定义
- 音名选择逻辑
- 根音映射逻辑

### 对当前需求的影响

对于”和弦库动态化”这件事，这个决策的好处是：
- 新模块的显示逻辑更简单
- 与数据源符号一致，减少转换
- 测试和调试更直观

因此建议把它作为**和弦库新模块开发前的轻量清理**写入实施计划。

## 17. 结论

“和弦图表模块动态化”的本质，不只是把图片换成 SVG。

它真正的价值在于：

1. 把和弦图从静态素材升级为结构化数据
2. 把和弦字典从不可扩展的图片库升级为可演进的训练基础设施
3. 为后续练耳、指板、和弦进行、调式应用等模块建立共享底座

从当前项目状态来看，这个需求非常值得优先推进，而且技术路径也已经比较清晰：

**共享渲染器 + 完整数据源精选 + 根音目录映射 + 动态浏览 UI**

这是当前最稳、最实用、最符合你产品方向的一条路线。
